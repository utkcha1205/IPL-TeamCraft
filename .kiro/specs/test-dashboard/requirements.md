# Requirements Document

## Introduction

The Test Dashboard is a new page in the IPL Player Stats app that displays test results from multiple test runners (vitest, playwright, jest) in a unified view. The dashboard parses JSON report files generated by each runner and presents per-runner statistics (pass count, fail count, total tests, duration), aggregate totals across all runners, and line coverage percentage. Test report data is sourced via a Next.js API route that reads JSON report files from a configurable directory.

## Glossary

- **Dashboard_Page**: The new `/test-dashboard` route that displays test results
- **Test_Report**: A JSON file generated by a test runner containing test results (pass/fail counts, duration, test names)
- **Runner_Card**: A UI card component that displays statistics for a single test runner
- **Aggregate_Summary**: A UI section that displays combined totals across all test runners
- **Coverage_Display**: A UI element that shows line coverage percentage from a coverage summary JSON file
- **Report_Parser**: A utility module that normalizes JSON report files from different test runners into a common internal format
- **Report_API**: A Next.js API route that reads test report JSON files from disk and returns normalized data
- **Runner_Type**: One of the supported test runner identifiers: "vitest", "playwright", or "jest"
- **Normalized_Result**: The common internal data structure representing test results regardless of the originating runner
- **TestCase**: An individual test entry with testName, status, durationMs, and optional errorMessage
- **File_Coverage**: Per-file coverage data including lines, statements, functions, and branches metrics
- **Component_Flow_Diagram**: A visual diagram showing the component architecture and data flow of the dashboard

## Requirements

### Requirement 1: Normalized Test Result Data Model

**User Story:** As a developer, I want a common data model for test results, so that the dashboard can display results from any supported runner uniformly.

#### Acceptance Criteria

1. THE Normalized_Result type SHALL include the following fields: runnerType (Runner_Type), totalTests (number), passCount (number), failCount (number), durationMs (number), and allTests (array of TestCase objects)
2. THE Normalized_Result type SHALL include an optional failedTests field containing an array of objects with testName (string) and errorMessage (string)
3. THE TestCase type SHALL include testName (string), status ("passed" or "failed"), durationMs (number), and an optional errorMessage (string)
4. FOR ALL valid Normalized_Result objects, passCount plus failCount SHALL equal totalTests (count invariant)

### Requirement 2: Report Parser — Vitest JSON Reports

**User Story:** As a developer, I want the Report_Parser to parse vitest JSON reporter output, so that vitest results appear on the dashboard.

#### Acceptance Criteria

1. WHEN a valid vitest JSON report file is provided, THE Report_Parser SHALL produce a Normalized_Result with runnerType set to "vitest"
2. WHEN a vitest JSON report contains test results, THE Report_Parser SHALL extract passCount from tests with status "passed" and failCount from tests with status "failed"
3. WHEN a vitest JSON report contains timing data, THE Report_Parser SHALL extract durationMs as the total duration in milliseconds
4. IF a vitest JSON report file is malformed or missing required fields, THEN THE Report_Parser SHALL return a descriptive error message
5. FOR ALL valid vitest JSON reports, parsing then serializing then parsing the Normalized_Result SHALL produce an equivalent object (round-trip property)

### Requirement 3: Report Parser — Playwright JSON Reports

**User Story:** As a developer, I want the Report_Parser to parse playwright JSON reporter output, so that playwright results appear on the dashboard.

#### Acceptance Criteria

1. WHEN a valid playwright JSON report file is provided, THE Report_Parser SHALL produce a Normalized_Result with runnerType set to "playwright"
2. WHEN a playwright JSON report contains test suites, THE Report_Parser SHALL recursively traverse suites to count passed and failed tests
3. WHEN a playwright JSON report contains timing data, THE Report_Parser SHALL compute durationMs from the report's start and end timestamps
4. IF a playwright JSON report file is malformed or missing required fields, THEN THE Report_Parser SHALL return a descriptive error message

### Requirement 4: Report Parser — Jest JSON Reports

**User Story:** As a developer, I want the Report_Parser to parse jest JSON reporter output, so that jest results appear on the dashboard.

#### Acceptance Criteria

1. WHEN a valid jest JSON report file is provided, THE Report_Parser SHALL produce a Normalized_Result with runnerType set to "jest"
2. WHEN a jest JSON report contains numPassedTests and numFailedTests fields, THE Report_Parser SHALL map those directly to passCount and failCount
3. WHEN a jest JSON report contains a startTime and an endTime or duration, THE Report_Parser SHALL extract durationMs in milliseconds
4. IF a jest JSON report file is malformed or missing required fields, THEN THE Report_Parser SHALL return a descriptive error message

### Requirement 5: Report Parser — Pretty Printer and Round-Trip

**User Story:** As a developer, I want a pretty printer that serializes Normalized_Result objects back to a canonical JSON format, so that round-trip consistency can be verified.

#### Acceptance Criteria

1. THE Report_Parser SHALL export a serialize function that converts a Normalized_Result into a JSON string
2. THE Report_Parser SHALL export a deserialize function that converts a JSON string back into a Normalized_Result
3. FOR ALL valid Normalized_Result objects, deserialize(serialize(result)) SHALL produce an object deeply equal to the original (round-trip property)

### Requirement 6: Coverage Data Parsing

**User Story:** As a developer, I want the Report_Parser to extract line coverage percentage from a coverage summary JSON file, so that the dashboard can display coverage.

#### Acceptance Criteria

1. WHEN a valid coverage summary JSON file is provided (e.g., coverage-summary.json), THE Report_Parser SHALL extract the total line coverage percentage
2. THE Report_Parser SHALL return the coverage value as a number between 0 and 100
3. IF the coverage summary file is missing or malformed, THEN THE Report_Parser SHALL return null instead of throwing an error
4. WHEN a valid coverage summary JSON file is provided, THE Report_Parser SHALL also extract per-file coverage data (lines, statements, functions, branches) via a parseFileCoverage function

### Requirement 7: Report API Route

**User Story:** As a developer, I want a Next.js API route that reads test report files and returns normalized results, so that the dashboard page can fetch data via HTTP.

#### Acceptance Criteria

1. THE Report_API SHALL be accessible at `/api/test-reports`
2. WHEN a GET request is made, THE Report_API SHALL read JSON report files from the `test-reports/` directory at the project root
3. WHEN report files are found, THE Report_API SHALL return a JSON response containing an array of Normalized_Result objects, a coveragePercent field, and a fileCoverage array
4. IF the `test-reports/` directory does not exist, THEN THE Report_API SHALL return an empty array for results, null for coveragePercent, and an empty array for fileCoverage
5. IF a report file fails to parse, THEN THE Report_API SHALL skip that file and include a warning in the response's warnings array

### Requirement 8: Dashboard Page — Layout and Navigation

**User Story:** As a user, I want to access the Test Dashboard from the main navigation, so that I can view test results alongside other app features.

#### Acceptance Criteria

1. THE Dashboard_Page SHALL be accessible at the `/test-dashboard` route
2. THE app header on the home page SHALL include a navigation link labeled "Test Dashboard" that navigates to `/test-dashboard`
3. THE Dashboard_Page SHALL use CSS variable inline styles consistent with the existing app theme (no Tailwind dark: classes)
4. THE Dashboard_Page SHALL include a page title "Test Dashboard" displayed prominently at the top

### Requirement 9: Dashboard Page — Runner Cards

**User Story:** As a user, I want to see a summary card for each test runner, so that I can quickly assess the health of each test suite.

#### Acceptance Criteria

1. WHEN test results are loaded, THE Dashboard_Page SHALL display one Runner_Card for each Runner_Type present in the data
2. THE Runner_Card SHALL display the runner name, passCount, failCount, totalTests, and durationMs formatted as human-readable text
3. WHEN all tests in a runner pass, THE Runner_Card SHALL display a green success indicator
4. WHEN any tests in a runner fail, THE Runner_Card SHALL display a red failure indicator with the fail count
5. THE Runner_Card SHALL display duration formatted as seconds with one decimal place (e.g., "2.3s")
6. THE Dashboard_Page SHALL render Runner_Cards in a responsive grid layout: 1 column on mobile, 2 columns on tablet, 3 columns on desktop

### Requirement 10: Dashboard Page — Aggregate Summary

**User Story:** As a user, I want to see aggregate totals across all test runners, so that I can understand overall test health at a glance.

#### Acceptance Criteria

1. THE Aggregate_Summary SHALL display totalTests, totalPass, and totalFail summed across all runners
2. THE Aggregate_Summary SHALL display an overall pass rate as a percentage (totalPass / totalTests * 100), rounded to one decimal place
3. WHEN totalTests is zero, THE Aggregate_Summary SHALL display "No test data available" instead of computed statistics
4. THE Aggregate_Summary SHALL be positioned above the Runner_Cards section

### Requirement 11: Dashboard Page — Coverage Display

**User Story:** As a user, I want to see line coverage percentage displayed prominently, so that I can track code coverage at a glance.

#### Acceptance Criteria

1. THE Coverage_Display SHALL show the line coverage percentage as a large number with a "%" suffix
2. WHEN coverage is 80% or above, THE Coverage_Display SHALL use a green color accent
3. WHEN coverage is between 50% and 79%, THE Coverage_Display SHALL use a yellow color accent
4. WHEN coverage is below 50%, THE Coverage_Display SHALL use a red color accent
5. WHEN coverage data is unavailable, THE Coverage_Display SHALL display "No coverage data" with a muted style

### Requirement 12: Dashboard Page — Failed Tests Detail

**User Story:** As a user, I want to see a list of failed tests with error messages, so that I can quickly identify what needs fixing.

#### Acceptance Criteria

1. WHEN any runner has failed tests, THE Dashboard_Page SHALL display a "Failed Tests" section below the Runner_Cards
2. THE "Failed Tests" section SHALL list each failed test with its testName, errorMessage, and the Runner_Type it belongs to
3. WHEN no tests have failed across all runners, THE Dashboard_Page SHALL hide the "Failed Tests" section entirely

### Requirement 13: Aggregate Computation Utility

**User Story:** As a developer, I want a utility function that computes aggregate statistics from an array of Normalized_Result objects, so that the computation is testable independently of the UI.

#### Acceptance Criteria

1. WHEN given an array of Normalized_Result objects, THE utility SHALL return an object with totalTests, totalPass, totalFail, and passRate fields
2. FOR ALL valid arrays of Normalized_Result objects, totalPass plus totalFail in the aggregate SHALL equal totalTests (count invariant)
3. FOR ALL valid arrays, the aggregate totalTests SHALL equal the sum of each individual result's totalTests (additive property)
4. WHEN given an empty array, THE utility SHALL return totalTests as 0, totalPass as 0, totalFail as 0, and passRate as 0

### Requirement 14: Error Handling

**User Story:** As a user, I want the Dashboard_Page to handle errors gracefully, so that the page remains usable even when test data is unavailable.

#### Acceptance Criteria

1. IF the Report_API returns an error, THEN THE Dashboard_Page SHALL display a user-friendly error message "Failed to load test results"
2. IF the Report_API returns an empty results array, THEN THE Dashboard_Page SHALL display "No test reports found. Run your test suites to generate reports."
3. WHILE the Dashboard_Page is fetching data from the Report_API, THE Dashboard_Page SHALL display a loading indicator

### Requirement 15: Dashboard Page — All Tests Tab with Filters

**User Story:** As a user, I want to browse all individual tests across runners with filtering capabilities, so that I can find specific tests quickly.

#### Acceptance Criteria

1. THE Dashboard_Page SHALL include a tabbed interface with "All Tests", "File Coverage", and "Component Flow" tabs
2. THE "All Tests" tab SHALL list every individual test case from all runners, showing test name, status (PASS/FAIL), runner type, inferred test type, duration, and error message if applicable
3. THE "All Tests" tab SHALL provide filter dropdowns for runner type, test status, and test type
4. THE Dashboard_Page SHALL infer test types (unit, integration, e2e, property) from runner type and test name patterns

### Requirement 16: Dashboard Page — File Coverage Tab

**User Story:** As a user, I want to see per-file coverage metrics, so that I can identify which files need more test coverage.

#### Acceptance Criteria

1. THE "File Coverage" tab SHALL display a table of per-file coverage data with columns for File, Lines, Statements, Functions, and Branches
2. EACH coverage metric SHALL display the percentage with color coding (green ≥ 80, yellow 50–79, red < 50) and the covered/total count
3. WHEN no file coverage data is available, THE tab SHALL display "No file coverage data available. Generate a coverage report to see per-file metrics."

### Requirement 17: Dashboard Page — Component Flow Tab

**User Story:** As a user, I want to see a visual diagram of the dashboard's component architecture, so that I can understand the data flow.

#### Acceptance Criteria

1. THE "Component Flow" tab SHALL render a Component_Flow_Diagram showing the architecture and data flow of the test dashboard
